# TypeSpec Contract Design For Call Calendar

## Goal

Define a TypeSpec API contract that captures the MVP behavior of the booking application for two semantic roles:

- owner
- guest

The contract is the source of truth for system behavior. It must describe only the behavior already supported by the product description in `README.md` and must not introduce registration, authentication, or extra user roles.

## Scope

This design covers:

- event type creation by the owner
- event type listing
- available slot lookup for a selected event type
- booking creation by a guest
- booking cancellation
- owner access to the full booking list

This design does not cover:

- registration
- authentication
- owner profile management
- event type updates or deletion
- booking rescheduling
- notifications
- payments

## Design Principles

- The API contract remains behavior-first and does not expose internal storage design.
- Public API slots are time intervals, not stored slot resources.
- Owner and guest capabilities are separated semantically in the contract, not through security schemes.
- Booking history is preserved by status changes instead of deleting booking records.

## Role Model

The system has two product roles:

- owner: the fixed calendar owner
- guest: any anonymous user booking a slot

The contract does not define a technical access-control mechanism for these roles. Instead, operations are documented and grouped as owner-facing or guest-facing behavior. This is an intentional MVP constraint aligned with the current product description.

## API Shape

Recommended API structure:

- `POST /event-types`
- `GET /event-types`
- `GET /event-types/{eventTypeId}/availability`
- `POST /bookings`
- `GET /bookings`
- `POST /bookings/{bookingId}:cancel`

This shape keeps core resources explicit while treating availability as a derived view of an event type and cancellation as a state transition on an existing booking.

## Resource Models

### EventType

Represents a bookable event configuration created by the owner.

Fields:

- `id`
- `title`
- `description?`
- `durationMinutes`

Rules:

- `title` is required.
- `durationMinutes` is required.
- `description` is optional.
- `durationMinutes` must be a positive integer.

### AvailableSlot

Represents a free time interval available for booking for a specific event type.

Fields:

- `startAt`
- `endAt`

Rules:

- Slots are generated by the system.
- Slots returned by the API are already aligned to the selected event type duration.
- The API exposes only free slots.

### Booking

Represents a guest reservation for a selected event type and time interval.

Fields:

- `id`
- `eventTypeId`
- `startAt`
- `endAt`
- `guestName`
- `guestEmail`
- `status`

Rules:

- `guestName` is required.
- `guestEmail` is required.
- `status` is controlled by the system.

### BookingStatus

Allowed values:

- `active`
- `cancelled`

## Operations

### Create Event Type

`POST /event-types`

Owner behavior:

- creates a new event type

Request body:

- `title`
- `description?`
- `durationMinutes`

Response:

- `201 Created` with created `EventType`

Errors:

- `400 Bad Request` for invalid payload structure or invalid duration

### List Event Types

`GET /event-types`

Guest and owner behavior:

- returns the list of available event types

Response:

- `200 OK` with a list of `EventType`

### Get Availability

`GET /event-types/{eventTypeId}/availability`

Guest behavior:

- returns free time intervals for the selected event type within the next 14 days

Response:

- `200 OK` with a list of `AvailableSlot`

Errors:

- `404 Not Found` if the event type does not exist

Note:

- The MVP contract does not require date-range query parameters for availability.
- The operation returns free slots for the next 14 days only.
- The contract must not introduce a separate slot resource.

### Create Booking

`POST /bookings`

Guest behavior:

- creates a booking for a selected event type and available interval

Request body:

- `eventTypeId`
- `startAt`
- `endAt`
- `guestName`
- `guestEmail`

Response:

- `201 Created` with created `Booking`

Errors:

- `400 Bad Request` for invalid payload structure or malformed timestamps
- `404 Not Found` if the event type does not exist
- `409 Conflict` if the interval is unavailable, outside the booking window, or already booked

### List Bookings

`GET /bookings`

Owner behavior:

- returns the full booking list

Response:

- `200 OK` with a list of `Booking`

Notes:

- The list includes both `active` and `cancelled` bookings.

### Cancel Booking

`POST /bookings/{bookingId}:cancel`

Owner and guest behavior:

- changes booking status from `active` to `cancelled`

Response:

- `200 OK` with updated `Booking`

Errors:

- `404 Not Found` if the booking does not exist
- `409 Conflict` if the booking is already cancelled

Notes:

- Cancellation does not delete the booking.
- After cancellation, the interval becomes available again.
- The contract intentionally does not define how the system technically verifies whether the caller is the owner or the guest who created the booking.

## Business Rules

The TypeSpec contract must make the following rules explicit:

1. Booking is limited to the next 14 days from the current moment.
2. Only free slots can be booked.
3. The same time interval cannot be booked twice, even across different event types.
4. Booking intervals must match the duration of the selected event type.
5. Availability is derived dynamically from current event types, current time, and existing active bookings.
6. Cancelled bookings remain visible in owner booking history.
7. Cancelling a booking makes its interval available again.

## Error Model

The contract should use a small, consistent error set:

- `400 Bad Request` for structurally invalid input
- `404 Not Found` for missing event types or bookings
- `409 Conflict` for business-rule violations

Recommended `409` cases:

- slot already booked
- slot outside booking window
- interval does not match current availability
- booking already cancelled

`422 Unprocessable Content` is intentionally excluded from the MVP contract to keep error handling simple and predictable.

## TypeSpec Modeling Guidance

When drafting the actual TypeSpec files:

- group operations by semantic role or resource namespace
- express owner-only and guest-facing behavior through naming, tags, and descriptions
- model date-time values explicitly and consistently
- keep booking status as an enum
- avoid introducing security schemes unless product requirements change
- avoid introducing slot IDs or a standalone slot resource

## Open Point Resolved

Guest cancellation is intentionally specified without a technical proof-of-ownership mechanism. This is not an omission; it is a conscious MVP simplification to preserve alignment with the current product description, which excludes authentication and does not define cancellation tokens.

If future product requirements demand stronger cancellation control, the contract can be extended later with a dedicated verification mechanism.

## Acceptance Criteria For The Contract

The resulting TypeSpec contract is acceptable if it:

- separates owner and guest behavior clearly
- captures all scenarios from `README.md`
- keeps the booking window at 14 days
- enforces cross-event slot uniqueness
- models cancellation as a status transition
- avoids adding undocumented product behavior

## Implementation Impact

Once approved, this design should drive:

- TypeSpec contract authoring in `spec/`
- backend behavior aligned to contract rules
- frontend flows aligned to contract operations and model shapes

No backend or frontend behavior should be implemented unless it is reflected in the final TypeSpec contract.
